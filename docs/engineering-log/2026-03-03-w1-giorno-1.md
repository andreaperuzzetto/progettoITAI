# W1 — Giorno 1  
Data: 2026-03-03  

## Obiettivo della giornata

Definire le fondamenta architetturali del progetto, impostando un monorepo strutturato e un backend modulare pronto per evoluzioni successive, senza integrare ancora servizi esterni.

---

## Contesto architetturale

La Settimana 1 è dedicata alla costruzione dell’ossatura tecnica del sistema.  
L’obiettivo non è sviluppare feature, ma stabilire una struttura solida, estendibile e coerente con una futura evoluzione SaaS multi-tenant con integrazione AI.

Una base disordinata in questa fase genererebbe debito tecnico nelle settimane successive.

---

## Decisioni architetturali

È stato adottato un monorepo gestito tramite pnpm workspaces.  
Questa scelta consente di mantenere frontend e backend nello stesso repository, garantendo coerenza strutturale e possibilità di estensione futura con package condivisi, senza introdurre complessità prematura come Turborepo o Nx.

Il backend è stato sviluppato con FastAPI, privilegiando un’architettura modulare a layer distinti (api, core, db, services).  
Questa separazione riduce accoppiamento e migliora manutenibilità e scalabilità futura.

La configurazione è stata centralizzata tramite pydantic-settings, evitando variabili hardcoded e preparando il sistema alla gestione multi-environment.

È stato introdotto un controllo di connessione al database in fase di startup per implementare una strategia fail-fast, evitando errori silenziosi a runtime.

---

## Implementazione tecnica

È stata creata la seguente struttura:

progettoITAI/  
├── apps/  
│   ├── frontend/  
│   └── backend/  
├── docs/  

Nel backend:

app/  
├── main.py  
├── api/  
├── core/  
├── db/  
└── services/  

Sono stati implementati:

- configurazione centralizzata  
- logging base  
- health endpoint  
- SQLAlchemy engine e session factory  
- dependency injection per il database  
- startup lifecycle con test connessione  

Per validazione architetturale è stato utilizzato SQLite locale temporaneo.

---

## Problemi incontrati

Sono stati riscontrati errori relativi alla struttura delle cartelle e al comando uvicorn, dovuti a un posizionamento errato dei moduli.  
Il problema è stato risolto riallineando la gerarchia delle directory sotto il package `app`.

È stato inoltre riscontrato un comportamento di zsh relativo ai caratteri speciali durante l’installazione di uvicorn[standard], risolto utilizzando le virgolette nel comando pip.

## Stato finale del sistema

Il sistema dispone ora di:

- monorepo funzionante  
- frontend operativo  
- backend modulare  
- configurazione centralizzata  
- logging attivo  
- database layer pronto  
- health endpoint funzionante  

Non sono ancora integrate connessioni reali a Supabase, OpenAI o Pinecone.

## Implicazioni per i prossimi step

La struttura attuale consente di integrare Supabase senza modificare l’architettura interna.  
Il layer database è pronto per l’introduzione di modelli e migrazioni.  
La modularità predisposta riduce il rischio di refactoring nelle settimane successive.
