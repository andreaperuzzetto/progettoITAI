# W1 — Giorno 2
Data: 2026-03-04

## Obiettivo della giornata

Integrare i servizi esterni fondamentali del backend, consolidando l’infrastruttura tecnica prima dell’introduzione di modelli di dominio e logica applicativa.

La giornata è stata dedicata al cablaggio reale dei tre pilastri del sistema:

- Persistence Layer (PostgreSQL)
- Intelligence Layer (OpenAI)
- Vector Layer (Pinecone)

Non sono state sviluppate feature funzionali.

---

## Contesto architetturale

Dopo la costruzione dell’ossatura modulare nel Giorno 1, il focus è stato spostato sull’attivazione concreta delle dipendenze esterne.

L’obiettivo è stato rendere il backend capace di:

- connettersi a un database reale
- inizializzare un client AI
- predisporre un vector database

La logica seguita è stata sequenziale: prima configurazione stabile, poi connessione database, infine integrazione AI.

---

## Configurazione estesa

È stata estesa la configurazione centralizzata tramite `pydantic-settings`.

Sono state aggiunte variabili per:

- database_url
- openai_api_key
- pinecone_api_key
- pinecone_environment
- pinecone_index_name

La configurazione è ora il punto unico di accesso ai servizi esterni.

È stato mantenuto il principio:

nessuna variabile hardcoded.

---

## Attivazione PostgreSQL

È stato attivato PostgreSQL 18 locale.

Operazioni effettuate:

- configurazione PATH per accesso a `psql`
- creazione database `progettoitai`
- correzione stringa di connessione
- eliminazione credenziali placeholder errate
- verifica ruolo utente corretto

È stata validata la connessione tramite endpoint tecnico:

/debug/db-check

Implementazione SQLAlchemy:

- create_engine
- SessionLocal
- dependency injection get_db
- pool_pre_ping=True

Il persistence layer è ora operativo.

Non sono ancora presenti:

- modelli ORM
- schema relazionale
- migrazioni

---

## Integrazione OpenAI

È stato creato un service layer dedicato:

services/openai_client.py

Principi adottati:

- nessuna logica AI nei router
- client inizializzato tramite configurazione
- separazione tra API layer e AI layer

Endpoint tecnico introdotto:

/debug/openai-check

L’integrazione è strutturalmente corretta.  
L’assenza di API key reale non influisce sulla validità architetturale.

Non è stato ancora implementato:

- structured output
- enforcement JSON schema
- decision logging

---

## Integrazione Pinecone

È stato inizializzato il client Pinecone.

Funzionalità attuale:

- connessione tramite API key
- index configurabile
- endpoint tecnico di verifica

Non è ancora presente:

- pipeline embedding
- chunking
- retrieval API
- RAG engine

Il vector layer è predisposto ma non ancora utilizzato operativamente.

---

## Debugging e stabilizzazione

Durante la giornata sono stati risolti:

- errori di import path del package
- mancata installazione driver psycopg
- problemi PATH per psql
- ruolo database inesistente
- stringa connessione errata
- incoerenza variabili environment
- errore autenticazione OpenAI

Questa fase ha consolidato la stabilità dell’infrastruttura.

Il sistema ora fallisce in modo esplicito in caso di dipendenze non configurate.

---

## Stato del sistema a fine giornata

Il backend dispone di:

- configurazione centralizzata estesa
- PostgreSQL connesso e verificato
- SQLAlchemy operativo
- service layer AI isolato
- client Pinecone predisposto
- endpoint tecnici di verifica

Non sono ancora presenti feature applicative né modelli di dominio.

Lo stato del sistema può essere definito:

“Infrastrutturalmente completo, funzionalmente minimale.”
